# 变电站巡检仿真系统 — 设计文档

> 版本：2026-06-12  
> 架构：黑板模式 + MQ 消息总线  
> 团队：4 人（Person A/common+controller / Person B/car / Person C/navigator+target-planner+task-configurator / Person D/display+launcher）

## 一、系统架构

```
                     ┌──────────────────┐
                     │     Redis 黑板    │ ← 共享状态
                     │  数据字典 9 类 key │
                     └──┬──┬──┬──┬──┬──┘
                        │  │  │  │  │
    ┌───────────────────┼──┼──┼──┼──┼───────────────────┐
    │                   │  │  │  │  │                   │
    ▼                   ▼  ▼  ▼  ▼  ▼                   ▼
┌─────────┐        ┌─────────────────────┐         ┌─────────┐
│Car × N  │◄──MQ──►│    Controller       │◄──MQ──►│Display  │
│知识源    │        │    TickScheduler    │         │WebSocket│
│5状态机   │        │    StatusDispatcher │         │+ HTTP   │
└─────────┘        │    CommandHandler   │         └────┬────┘
                   └──────┬──┬──┬───────┘              │
                          │  │  │                  浏览器
            ┌─────────────┘  │  └─────────────┐
            ▼                ▼                ▼
      ┌──────────┐   ┌──────────┐    ┌────────────────┐
      │Navigator │   │Target    │    │TaskConfigurator│
      │BFS/A*    │   │Planner   │    │初始化/重置      │
      └──────────┘   │贪心分配   │    └────────────────┘
                     └──────────┘
```

**核心原则**：
- 知识源之间不直接通信，全部通过 Controller 中转
- 状态数据只存在 Redis 黑板中，无本地缓存
- 每个模块独立进程，通过 MQ 异步消息驱动

## 四、Controller 调度算法

### 4.1 状态机（5 种状态）

```
IDLE ──ASSIGN_TARGET──► WAITING_ROUTE
                          │ hasTarget=yes
                          ▼
                      PLAN_ROUTE ──► READY ──TICK_MOVE──► MOVING
                                      ▲                    │
                                      │    有下一步 ◄──────┘
                                      │
                                   路径走完 → IDLE
                                      │
                                   有障碍 → BLOCKED ──超时 2 tick──► IDLE
```

**状态写入权限**：
| 状态变迁 | 写入者 |
|----------|--------|
| → IDLE | Car（路径走完）、Controller（受阻超时、路线失败） |
| → WAITING_ROUTE | Controller（收到 TARGET_ASSIGNED） |
| → READY | Controller（收到 ROUTE_PLANNED）、Car（移动后路径非空） |
| → MOVING | Car（收到 TICK_MOVE） |
| → BLOCKED | Car（下一步有障碍） |

### 4.2 dispatch() 流程

```
每 tick:
  ├─ 任务激活？→ 否 → 跳过
  ├─ 探索率 ≥ 98%？→ 是 → completeTask()
  ├─ 发现车辆（KEYS Car*:Status）
  ├─ 第一轮：按状态分发
  │   ├─ IDLE → sendAssignTarget（发到 TargetPlanner）
  │   ├─ WAITING_ROUTE → checkAndPlanRoute（有目标就发到 Navigator）
  │   ├─ MOVING → checkMovingStuck（≥2 tick 切 READY）
  │   └─ BLOCKED → checkBlockedTimeout（≥2 tick 切 IDLE）
  ├─ 第二轮：READY 车发 TICK_MOVE
  └─ broadcastRefresh → REFRESH_ALL 广播
```

## 五、路径规划（Navigator）

### 5.1 BFS / A*

| 算法 | 特点 |
|------|------|
| BFS | 队列 + 4 方向扩展，保证最短路径，O(W×H) |
| A* | 优先队列 + 曼哈顿启发式，更快收敛 |

### 5.2 路径存储方向

Navigator 返回 `[step1, step2, ..., target]`（不含起点）。

**写入**：`pushRoute` 逆序 LPUSH → Redis 列表 `[step1, step2, ..., target]`  
**读取**：`LINDEX 0`（peek 下一步）、`LPOP`（消费下一步）、`LRANGE 0 -1`（Display 读取完整路线）

### 5.3 避让策略

BFS/A* 规划时预读 `mapBlock`（静态障碍物）+ 所有车的 `CarID:Position`，合并为 blocked 矩阵。车占位不写入 `mapBlock`。

## 六、目标分配（TargetPlanner）

### 6.1 最远优先策略

```
1. 收集所有"未探索且非障碍"的格子
2. 排除本 tick 已分配给其他车的
3. 按曼哈顿距离降序排列
4. 取最远的一个
```

目的：让多辆车分散到地图不同角落，避免路径交叉互堵。

## 七、任务生命周期

```
初始状态 ──[SET_CONFIG]──► TaskConfigurator.initialize()
                              ├─ FLUSHDB 清空
                              ├─ 写 TaskConfig
                              ├─ 随机障碍物 → mapBlock
                              ├─ 初始车位 → CarID:Position
                              ├─ 3×3 点亮 → mapView
                              └─ 发 TASK_READY → Controller
                                                      │
                                              TickScheduler.start()
                                                      │
                                              每 tick 调度 ←──┐
                                                      │       │
                                              [暂停] ──┘       │
                                                      │       │
                                              [重置] ──► FLUSHDB（等新开始）
                                                      │
                                         探索率 ≥ 98% │
                                                      ▼
                                              completeTask()
                                              ├─ 停止 tick
                                              ├─ 计算耗时
                                              └─ 最后一帧广播 → 前端显示"✓ 任务完成"
```

## 八、防冲突机制

### 8.1 分布式锁（Car 自身）

`DistributedLock`：`lock:{carId}`，SET NX PX 5000，Lua 脚本释放。保证同一 Car 不被并发操作。

### 8.2 位置预约锁（Car 间）

`tryReservePosition(x, y, carId)`：`pos:reserve:{x}:{y}`，SET NX EX 3。移动前预约目标格——成功则执行移动，失败则退回 READY 等下一 tick。消除多车同时移动到同一格。
