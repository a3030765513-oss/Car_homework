### 2.3 黑板数据字典初始化

每个模块启动时不负责初始化黑板数据，**统一由 TaskConfigurator 负责**。

TaskConfigurator 收到 `FORWARD_CONFIG` 后执行初始化：

1. `FLUSHDB`（或按 key 逐个清理）
2. 随机生成障碍物到 `mapBlock`（按 `obstacleRatio`）
3. 初始化 `carCount` 台车位置到 `CarID:Position`（随机不重叠、非障碍）
4. 设置 `CarID:Status` = `IDLE`
5. 设置 `CarID:Steps` = `0`
6. 设置 `TaskConfig` 完整配置
7. 各车初始位置点亮周围 3×3 到 `mapView`
8. 通知 Controller `TASK_READY`

## 三、阶段 2：核心逻辑开发（3-4 天）

### 3.1 Controller 模块

> ⚠️ **任务书约束**：控制器必须独立运行，且在操作系统中只能运行一个实例。ControllerMain 启动时必须检测是否已有实例在运行（通过 Redis 设置 `controller:instance` key + SETNX 互斥），若已有实例则报错退出。

**核心类**：

| 类名 | 职责 |
|------|------|
| `ControllerMain` | 入口，启动调度器 |
| `TickScheduler` | 定时器驱动 tick()，间隔 = TaskConfig.tickInterval（默认 500ms） |
| `StatusDispatcher` | 遍历所有已注册车辆状态，分发命令 |
| `CommandHandler` | 处理 ControllerCmd 队列中的回复消息 |

**tick() 主流程**：

```java
void tick() {
    // 1. 检查任务是否激活
    if (!blackboard.isTaskActive()) return;

    // 2. 完成判定：探索率 ≥ 99.9%
    if (blackboard.getExplorationRate() >= 99.9) {
        taskComplete();
        return;
    }

    // 3. 状态调度：遍历所有小车
    for (String carId : carIds) {
        CarStatus status = blackboard.getCarStatus(carId);
        switch (status) {
            case IDLE:
                // 向 TargetPlanner 发 ASSIGN_TARGET
                messageBus.publish("TargetPlannerCmd", buildAssignTargetMsg(carId, tick));
                pendingTargetRequests.add(carId);
                break;
            case WAITING_ROUTE:
                // 检查是否已有目标，有则向 Navigator 发 PLAN_ROUTE
                if (blackboard.hasTarget(carId)) {
                    Point target = blackboard.getCarTarget(carId);
                    Point pos = blackboard.getCarPosition(carId);
                    messageBus.publish("NavigatorCmd", buildPlanRouteMsg(carId, pos, target, tick));
                }
                break;
            case READY:
                // 跳过，等 TICK_MOVE
                break;
            case MOVING:
                // 异常恢复：重置为 READY（上一拍没收到回复的情况）
                blackboard.setCarStatus(carId, CarStatus.READY);
                break;
            case BLOCKED:
                // 检查受阻超时：当前 tick - blockedTick ≥ 2
                int blockedTick = blackboard.getBlockedTick(carId);
                if (tick - blockedTick >= 2) {
                    blackboard.clearRoute(carId);
                    blackboard.clearCarTarget(carId);
                    blackboard.clearBlockedTick(carId);
                    blackboard.setCarStatus(carId, CarStatus.IDLE);
                    messageBus.publish("Car_" + carId, buildBlockedTimeoutMsg(carId, tick));
                }
                break;
        }
    }

    // 4. 节拍移动：向所有 READY 车发 TICK_MOVE
    for (String carId : carIds) {
        if (blackboard.getCarStatus(carId) == CarStatus.READY) {
            messageBus.publish("Car_" + carId, buildTickMoveMsg(carId, tick));
        }
    }

    // 5. 广播刷新
    messageBus.publishFanout("UpdateView", buildRefreshAllMsg(tick));

    // 6. 处理 Web 命令（在 CommandHandler 中异步处理）
}
```

**Controller 处理知识源回复**（在 CommandHandler 中）：

| 收到消息 | 处理 |
|----------|------|
| `TARGET_ASSIGNED` | 若 carId 在 pendingTargetRequests 中且 target != null，写 `CarID:Status` = `WAITING_ROUTE`；从 pendingTargetRequests 移除 |
| `ROUTE_PLANNED` | 若 routeFound=true，写 `CarID:Status` = `READY` |
| `MOVED` | 无需额外处理（Car 已自己写了状态） |
| `ROUTE_DONE` | 记录完成事件 |
| `BLOCKED` | 记录受阻事件，后续 tick 检查超时 |
| `TASK_READY` | 启动 tick 循环 |
| `SET_CONFIG` | 转发 `FORWARD_CONFIG` 到 TaskConfigurator |
| `RESET` | 转发 `FORWARD_RESET` 到 TaskConfigurator |

### 3.2 Car 模块

> 📌 **动态添加小车**：任务书要求"用户可独立启动、添加新的小车参加地图任务探索"。由于每个 Car 实例是独立 Java 进程、通过 carId 参数标识自身，启动新 Car 进程（如 `CarMain Car006`）即自动注册到黑板并参与任务。Controller 和 TargetPlanner 通过 Redis `KEYS Car*:Status` 动态获取已注册车辆列表（而非从 carCount 硬推导），新 CarMain 启动即自动加入。

**核心类**：

| 类名 | 职责 |
|------|------|
| `CarMain` | 入口，连接 Redis + 订阅 Car_CarID 队列 |
| `CarAgent` | 5 状态状态机 + TICK_MOVE 处理 |
| `MoveExecutor` | 移动一步的 8 步原子操作 |

**CarAgent 消息处理**：

```
收到 TICK_MOVE:
  1. 用分布式锁 lock:CarID 加锁
  2. 读当前状态，若非 READY 则解锁返回
  3. peekNextRouteStep → 获取下一步位置
  4. 检查 mapBlock → 有障碍则：
     a. 清空 RouteList
     b. 写 Status=BLOCKED
     c. 写 BlockedTick=当前tick
     d. 发 BLOCKED 给 ControllerCmd
     e. 解锁返回
  5. popNextRouteStep → 消费该步
  6. 清除旧位置 mapBlock 标记
  7. 更新 CarID:Position
  8. 设置新位置 mapBlock 标记
  9. illuminateArea → 点亮 3×3
  10. incrementCarSteps
  11. 判断 RouteList 是否为空：
      空 → 写 Status=IDLE，发 ROUTE_DONE
      非空 → 写 Status=READY，发 MOVED
  12. 解锁
```

**点亮 3×3 算法**：

```java
void illuminateArea(Point center, BlackboardClient bb, int mapWidth, int mapHeight) {
    for (int dr = -1; dr <= 1; dr++) {
        for (int dc = -1; dc <= 1; dc++) {
            int r = center.y + dr;
            int c = center.x + dc;
            if (r >= 0 && r < mapHeight && c >= 0 && c < mapWidth) {
                bb.setMapViewBit(r, c, true);
            }
        }
    }
}
```

### 3.3 Navigator 模块

**核心类**：

| 类名 | 职责 |
|------|------|
| `NavigatorMain` | 入口，订阅 NavigatorCmd |
| `PathPlanner` | 接口：`List<Point> plan(Point start, Point target, BlackboardClient bb)` |
| `BfsPathFinder` | BFS 实现，4 方向扩展 |
| `AStarPathFinder` | A* 实现，曼哈顿距离启发 |
| `PathPlannerFactory` | 根据 algorithm 参数返回对应实现 |

**BFS 算法要点**：
- 4 方向：上下左右
- 避开 `mapBlock` 中的障碍 + 其他车辆的当前位置
- 边界检查
- 返回从 start 到 target 的完整路径（不含 start，含 target）

> 📌 **多实例支持**：NavigatorCmd 是 Classic Queue，多个 Navigator 进程可共享竞争消费，实现"分解导航压力"。验收：启动 2 个 Navigator 实例，RabbitMQ 管理界面可见多个消费者。

**A* 算法要点**：
- 启发函数：曼哈顿距离 `|dx| + |dy|`
- 开放列表用 `PriorityQueue`
- 关闭列表用 `boolean[][] visited`

### 3.4 TargetPlanner 模块

**核心类**：

| 类名 | 职责 |
|------|------|
| `TargetPlannerMain` | 入口，订阅 TargetPlannerCmd |
| `GreedyTargetAllocator` | 贪心目标分配算法 |

**贪心分配算法**：

```java
Point allocateTarget(String carId, Point currentPos, BlackboardClient bb,
                     Set<Point> alreadyAllocated, int mapWidth, int mapHeight) {
    // 1. 收集所有未探索 + 非障碍的格子
    List<Point> unexplored = new ArrayList<>();
    for (int r = 0; r < mapHeight; r++) {
        for (int c = 0; c < mapWidth; c++) {
            if (!bb.getMapViewBit(r, c) && !bb.isBlocked(r, c)) {
                Point p = new Point(c, r);
                if (!alreadyAllocated.contains(p)) {
                    unexplored.add(p);
                }
            }
        }
    }

    if (unexplored.isEmpty()) return null;

    // 2. 按曼哈顿距离排序
    unexplored.sort(Comparator.comparingInt(p -> p.manhattanDistance(currentPos)));

    // 3. 距离规则
    if (unexplored.size() > 1) {
        // 剩余多个未探索格子：只分配距离 ≥ 10 的
        List<Point> farEnough = unexplored.stream()
            .filter(p -> p.manhattanDistance(currentPos) >= 10)
            .collect(Collectors.toList());
        if (!farEnough.isEmpty()) {
            Point chosen = farEnough.get(0);
            alreadyAllocated.add(chosen);
            return chosen;
        }
        // 没有距离 ≥ 10 的，暂不分配
        return null;
    } else {
        // 最后一个格子：无距离限制
        Point chosen = unexplored.get(0);
        alreadyAllocated.add(chosen);
        return chosen;
    }
}
```

### 3.5 TaskConfigurator 模块

**初始化流程**：

```
收到 FORWARD_CONFIG(config):
  1. 清空所有 Redis key（FLUSHDB 或逐个 DEL）
  2. 解析配置：mapWidth, mapHeight, carCount, obstacleRatio, algorithm, tickInterval
  3. 写入 TaskConfig hash
  4. 随机生成障碍物到 mapBlock（避开小车初始位置）

     a. 随机选取非障碍空格子作为初始位置
     b. 写入 CarID:Position
     c. 写入 CarID:Status = IDLE
     d. 写入 CarID:Steps = 0
     e. 点亮该位置周围 3×3 到 mapView
  6. 发布 TASK_READY 到 ControllerCmd
```

## 六、关键技术决策记录

| # | 决策项 | 决定 | 理由 |
|---|--------|------|------|
| 1 | Controller 异步回复处理 | 同一 tick 内不等待 TP/Nav 回复；用 pendingTargetRequests 追踪，下一 tick 检查黑板状态 | 避免 tick 阻塞；TP/Nav 可能较慢 |
| 2 | 队列声明策略 | 各模块自声明（幂等） | 降低启动顺序依赖 |
| 3 | 节拍间隔默认值 | 500ms | 兼顾实时性和性能 |
| 4 | Car 移动加锁 | Redis 分布式锁 lock:CarID，锁超时 5s | 防并发写状态 |
| 5 | 路径存储方向 | LPUSH 写入（从头加），RPOP 读取（从尾取） | 这样 peek 用 LINDEX -1 看队首下一步 |
| 6 | 无目标时处理 | 删 key（DEL CarID:Target），不用 -1 | 语义清晰 |
| 7 | Controller 写 Status 时机 | 收到 TP/Nav 回复后立即写，不等到下个 tick | TP/Nav 不写 Status，Controller 统一控制 |
| 8 | 小车初始位置点亮 | TaskConfigurator 初始化时立即点亮各车周围 3×3 | 避免可视化时全部黑屏 |
| 9 | Controller 是否侦听 MQ | 订阅 ControllerCmd 接收知识源回复 | 原始设计要求"控制器不侦听队列，只发送"，但异步调度需要知识源通知 Controller 操作结果（TARGET_ASSIGNED / ROUTE_PLANNED 等）；Controller 只订阅 ControllerCmd 统一入口，不与任何特定知识源直接耦合 |

## 七、降级路径

如果某个模块的完整功能来不及开发，优先实现最小可行版本，确保端到端流程先跑通：

| 模块 | 最小可行版本（MVP） | 后续增强 |
|------|---------------------|---------|
| Navigator | 仅实现 BFS（4方向、避开障碍、边界检查） | 补充 A* + 启发函数 |
| Car | 不加分布式锁的移动（单 tick 单车移动） | 加 Redis 分布式锁 lock:CarID |
| TargetPlanner | 简单随机分配未探索格子（不做距离规则） | 贪心 + 距离 ≥ 10 规则 + 最后一格无限制 |
| Display | Canvas 最简渲染（只画地图 + 小车圆点） | 路径线 + 状态颜色 + 控制面板 + 图例 |
| Controller | tick 循环 + IDLE→TICK_MOVE + 单步移动 | 完整 5 状态分发 + pendingTargetRequests 追踪 |

**降级目标**：阶段 2a 结束时，必须能用 MVP 版本从初始化跑到探索率 ≥ 99.9%。

## 八、异常处理约定

分布式系统中各组件必须统一异常处理策略：

| 异常场景 | 处理方式 |
|----------|----------|
| MQ 连接断开 | RabbitMQ 自带自动恢复机制（`automaticRecoveryEnabled=true`），重连期间消息积压在队列中 |
| Redis 连接断开 | JedisPool 自动重试，超过 3 次失败则日志 ERROR + 进程退出（避免静默丢数据） |
| 收到格式错误的消息 | 日志 WARN + 丢弃该消息，不崩溃；格式：`[模块名] 收到非法消息: {原始内容}` |
| 分布式锁获取失败 | Car 跳过本次 TICK_MOVE，等下一节拍（不阻塞 tick 流程） |
| 路径未找到（Navigator） | 返回 routeFound=false，Controller 收到后将车状态置为 IDLE，等下一轮重试 |
| 目标未分配（TargetPlanner） | 返回 success=false，Controller 将车保持 IDLE，下一 tick 重新请求 |
| Controller 已有实例运行 | 新实例检测到 `controller:instance` 已存在 → 日志 ERROR 并 `System.exit(1)` |
