## 一、死锁修复

### 1.1 GreedyTargetAllocator 算法改为"最远优先"

**问题**：原算法"最近 ≥10"导致三辆车挤在同一片区域，路径交叉互相阻塞，集体卡在 IDLE。

**修复**：`selectByDistanceRule` 从"按距离升序 → 过滤 ≥10 → 回退最近"，改为"按距离降序 → 取最远"。车自然分散到地图不同角落，不再互堵。

**文件**：`target-planner/src/main/java/com/substation/targetplanner/GreedyTargetAllocator.java`

---

### 1.2 StatusDispatcher 阻塞超时时清空车占位（后回撤）

**问题**：车 BLOCKED 超时切回 IDLE 后，其 mapBlock 占位仍阻塞其他车的路径。

**修复（已回撤）**：在 `checkBlockedTimeout` 和 `finalizeMove` 中加 `setBlock(pos, false)`。

**回撤原因**：副作用是两车可穿行到同一格。后续因算法改为最远优先，死锁已解，不再需要此补丁。

## 二、探索率计算修复

### 2.1 公式改为逐格 GETBIT

**问题**：`getExplorationRate()` 用 `BITCOUNT(mapView)` / `BITCOUNT(mapBlock)` 计数，但位图偏移用的是构造函数 mapWidth（固定 30），而 `total` 用的是 TaskConfig 的 mapWidth。当前端配置不同尺寸时两者不一致，导致探索率显示错误（如 12%、100% 提前触发）。

**修复**：改为逐格 `GETBIT` 遍历，读写使用同一套偏移量（`row × 构造函数 stride + col`），`total` 读取 TaskConfig 实际尺寸，确保前后端计算一致。

**文件**：`common/src/main/java/com/substation/common/redis/BlackboardClient.java`

**修改前**：
```java
long total = (long) mapWidth * mapHeight;
long blocked = jedis.bitcount(KEY_MAP_BLOCK);
long explorable = total - blocked;
long explored = jedis.bitcount(KEY_MAP_VIEW);
return (int) (explored * 100 / explorable);
```

**修改后**：
```java
int w = 从 TaskConfig 读, h = 从 TaskConfig 读, 回退构造函数值;
for (int r = 0; r < h; r++)
    for (int c = 0; c < w; c++) {
        long offset = (long) r * mapWidth + c;  // 与 setMapViewBit 同偏移
        if (jedis.getbit(KEY_MAP_VIEW, offset)) explored++;
        if (jedis.getbit(KEY_MAP_BLOCK, offset)) blocked++;
    }
return (int) (explored * 100 / ((long) w * h - blocked));
```

## 三、完成判定强化

### 3.1 双机制触发 completeTask()

**问题**：某些场景下探索率永远达不到阈值（如被障碍物隔离的格子无法探索）。

**修复**：在 `StatusDispatcher.dispatch()` 中增加双重判定：
- **主判定**：探索率 ≥ 98%，立即完成
- **兜底判定**：全部车 IDLE 且无待处理请求，持续 30 tick → 强制完成

**文件**：`controller/src/main/java/com/substation/controller/StatusDispatcher.java`

### 3.2 completeTask() 补发最后一帧广播

**问题**：`completeTask()` 停掉 tick 后没发 REFRESH_ALL，前端不知道任务已结束。

**修复**：在 `completeTask()` 末尾广播一次 `explorationRate=100` 的 REFRESH_ALL。

## 四、前端修复

### 4.1 障碍物渲染跳过有车格子

**问题**：车移动时 mapBlock 标记其当前位置，前端 `renderObstacles()` 把该格也涂红，出现"小车拖红色方块"。

**修复**：渲染障碍物前先建 `occupied` 集合（遍历 `data.cars[].position`），有车的格子跳过不画。

**文件**：`display/src/main/resources/web/js/app.js`

### 4.2 WebSocket 重连后自动重置

**问题**：切浏览器标签页后 WebSocket 断开，重连时系统状态可能不一致导致卡死。

**修复**：`onSocketOpen` 检测到非首次连接（`wasEverConnected`）时，自动发送 `RESET` 消息重新初始化。

**文件**：`display/src/main/resources/web/js/app.js`

### 4.3 任务完成提示

**问题**：探索完成后无可视化提示。

**修复**：探索率 ≥99 时，顶栏 `info-mode` 标签显示 "✓ 任务完成"。

**文件**：`display/src/main/resources/web/js/app.js`

### 4.4 车辆数量输入只读 + 动态更新

**问题**：小车数量文本框可手改，与实际启动车辆数不一致。

**修复**：HTML `input` 加 `readonly` 属性，默认值改为 3；前端每帧从 `data.cars.length` 刷新该值。

**文件**：`display/src/main/resources/web/index.html`、`app.js`

## 五、动态添加小车修复

**问题**：前端按钮点击后无反应。

**修复**：
- `WebSocketBridge.onMessage()` 拦截 `ADD_CAR` 消息，用 `Runtime.exec(".\\mvnw.cmd exec:java -pl car ...")` 启动新车进程
- `CommandHandler.handle()` 增加 `TOGGLE_OBSTACLE` case，通过 `StatusDispatcher.toggleObstacle()` 切换障碍物

**文件**：`display/src/main/java/com/substation/display/WebSocketBridge.java`、`controller/src/main/java/com/substation/controller/CommandHandler.java`

## 六、Launcher 适配

**问题**：Launcher 仅 Display 使用构造函数传参，其余 5 个模块调 `main()` 硬编码 localhost，CLI 参数 `--redis-host` 等被忽略。

**修复**：NavigatorMain、TargetPlannerMain、TaskConfiguratorMain、CarMain 添加构造函数 `(redisHost, redisPort, mqHost, mqPort)` + `start()` 方法；LauncherMain 全部改为 `new XxxMain(...).start()`。

**文件**：`launcher/src/main/java/com/substation/launcher/LauncherMain.java`
