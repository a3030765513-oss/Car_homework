## 修复 1：ROUTE_PLANNED 失败导致车辆永久卡死

**严重程度**：高

### 原因

当 Navigator 找不到路径时，返回 `routeFound=false`。Controller 收到后，`onRoutePlanned` 方法对失败情况不做任何处理——车辆状态保持 `WAITING_ROUTE` 不变。下一个 tick，`StatusDispatcher` 看到车辆仍为 `WAITING_ROUTE`，再次向 Navigator 发送 `PLAN_ROUTE`。如果路径始终不可达，这会形成无限循环，车辆永远无法恢复。

设计文档明确要求：

> 路径未找到（Navigator）：返回 routeFound=false，Controller 收到后将车状态置为 IDLE，等下一轮重试

### 修改位置

`controller/src/main/java/com/substation/controller/StatusDispatcher.java`

### 修改前

```java
public void onRoutePlanned(String carId, boolean routeFound) {
    if (routeFound) {
        bb.setCarStatus(carId, CarStatus.READY);
    }
}
```

### 修改后

```java
public void onRoutePlanned(String carId, boolean routeFound) {
    if (routeFound) {
        bb.setCarStatus(carId, CarStatus.READY);
    } else {
        bb.setCarStatus(carId, CarStatus.IDLE);
    }
}
```

## 修复 2：Controller 单实例锁无过期时间

**严重程度**：中

### 原因

`acquireControllerLock()` 使用 `SETNX controller:instance "1"` 设置锁，没有设置 TTL（过期时间）。如果 Controller 进程被强制终止（`kill -9`、断电、蓝屏），JVM 的 shutdown hook 不会执行，`controller:instance` key 会永久残留在 Redis 中。此后任何新 Controller 实例都无法启动，必须手动登录 Redis 执行 `DEL controller:instance` 才能恢复。

### 修改位置

`common/src/main/java/com/substation/common/redis/BlackboardClient.java`

### 修改内容

**新增常量**：
```java
private static final int CONTROLLER_LOCK_TTL_SECONDS = 30;
```

**新增 import**：
```java
import redis.clients.jedis.params.SetParams;
```

### 修改前

```java
public boolean acquireControllerLock() {
    try (Jedis jedis = pool.getResource()) {
        return jedis.setnx(KEY_CONTROLLER_INSTANCE, "1") == 1L;
    }
}
```

### 修改后

```java
public boolean acquireControllerLock() {
    try (Jedis jedis = pool.getResource()) {
        SetParams params = SetParams.setParams().nx().ex(CONTROLLER_LOCK_TTL_SECONDS);
        return "OK".equals(jedis.set(KEY_CONTROLLER_INSTANCE, "1", params));
    }
}
```

### 说明

改用 `SET key value NX EX 30` 替代 `SETNX`。锁在 30 秒后自动过期。正常运行时 shutdown hook 会在进程退出时主动 `releaseControllerLock()`；异常崩溃时锁最多残留 30 秒，之后自动释放。

## 修复 3：魔法字符串 "BFS"

**严重程度**：中（违反 CLAUDE.md）

### 原因

硬编码字符串 `"BFS"` 作为默认算法类型。若 `AlgorithmType` 枚举值改名，此处静默失效。CLAUDE.md 明确禁止魔法字符串。

### 修改位置

`controller/src/main/java/com/substation/controller/StatusDispatcher.java`

### 修改内容

**新增 import**：
```java
import com.substation.common.model.AlgorithmType;
```

### 修改前

```java
FIELD_ALGORITHM, algorithm != null ? algorithm : "BFS"
```

### 修改后

```java
FIELD_ALGORITHM, algorithm != null ? algorithm : AlgorithmType.BFS.name()
```

## 修复 4：探索完成阈值 99.9 与整数类型不匹配

**严重程度**：低

### 原因

`EXPLORATION_COMPLETE` 声明为 `double = 99.9`，但 `getExplorationRate()` 返回值是 `int`（0~100 的整数）。Java 在比较 `int >= double` 时会将 int 提升为 double，所以 `99 >= 99.9` 为 false，`100 >= 99.9` 才为 true。这意味着任务永远在探索率达到 100% 才触发完成，而非设计文档要求的 99.9%。

将阈值改为 `int = 99`，当探索率达到 99% 即触发完成，更贴近设计意图。

### 修改位置

`controller/src/main/java/com/substation/controller/StatusDispatcher.java`

### 修改前

```java
private static final double EXPLORATION_COMPLETE = 99.9;
```

### 修改后

```java
private static final int EXPLORATION_COMPLETE = 99;
```

## 修复 5：节拍间隔校验中的魔法数字

**严重程度**：低（违反 CLAUDE.md）

### 原因

节拍间隔范围校验直接使用字面量 `100` 和 `2000`，缺乏语义表达。CLAUDE.md 要求提取为命名常量。

### 修改位置

`controller/src/main/java/com/substation/controller/CommandHandler.java`

### 修改前

```java
if (interval >= 100 && interval <= 2000) {
```

### 修改后

**新增常量**：
```java
private static final int MIN_TICK_INTERVAL_MS = 100;
private static final int MAX_TICK_INTERVAL_MS = 2000;
```

**校验语句**：
```java
if (interval >= MIN_TICK_INTERVAL_MS && interval <= MAX_TICK_INTERVAL_MS) {
```
