## 问题 1：如何在页面上动态添加小车

### 现状

目前 Car 模块通过命令行启动独立进程：
```powershell
.\mvnw.cmd exec:java -pl car "-Dexec.mainClass=com.substation.car.CarMain" "-Dexec.args=Car004"
```

CarMain 启动后自动在 Redis 注册（写入 `Car004:Status=IDLE` + `Car004:Position` 等），Controller 下一个 tick 通过 `KEYS Car*:Status` 自动发现新车并开始调度。

### 解决方案

**前端**（`display/src/main/resources/web/js/app.js`）：
- 在控制面板增加"添加小车"按钮
- 点击时通过 WebSocket 发送消息给 Display

**Display**（`WebSocketBridge.java` 的 `onMessage()`）：
- 接收到"添加小车"命令后，需要启动新的 CarMain 进程。由于 Java 无法在运行中动态创建新的 Maven 子进程（exec:java 需要 Maven 工具链），有两种方案：

| 方案 | 做法 | 优劣 |
|------|------|------|
| A | Launcher 启动时预创建 N 个空闲线程槽，前端"激活"某个槽 | 不需要动态创建进程，但槽数固定 |
| B | Display 收到命令后通过 `Runtime.getRuntime().exec("java -jar car.jar Car004")` 启动 | 需要先 `mvn package` 打包出 jar，但真正做到了动态添加 |

**推荐方案 B**。具体步骤：
1. `.\mvnw.cmd package -pl car -DskipTests` 生成 `car/target/car-1.0-SNAPSHOT.jar`
2. Display 的 `onMessage` 收到 `{type: "ADD_CAR"}` 时维护一个 carCount，执行：
   ```java
   String carId = String.format("Car%03d", ++carCount);
   Runtime.getRuntime().exec("java -jar car/target/car-1.0-SNAPSHOT.jar " + carId);
   ```
3. 前端按钮点击 → WebSocket 发送 `{type: "ADD_CAR"}`

## 问题 2：如何添加障碍物

### 现状

障碍物有两个来源：

1. **初始化阶段**（TaskConfigurator）：根据 `obstacleRatio`（默认 15%）随机生成到 `mapBlock` bitmap
2. **运行阶段**（DynamicObstacleUtil）：每 20 tick 随机增删 ≤2 个障碍物，避开车位

### 解决方案

**方式 A — 初始化时调整比例**：前端配置表单的"障碍物比例"滑块，范围 0~0.5（0%~50%）

**方式 B — 手动点击添加**（需前端改动）：Canvas 增加右键菜单，点击格子 → WebSocket 发送 `{type: "ADD_OBSTACLE", row: 12, col: 8}` → Display 转发 ControllerCmd → Controller 调用 `bb.setBlock(row, col, true)`

**当前最快做法**：在 `localhost:8887` 配置表单里调大障碍物比例，点"开始"即可看到更多红色块。

## 问题 3：重置后再点开始，系统不运行

### 原因

这是之前审查发现的**第 6 号中危问题**。

当前 `handleReset` 的流程：
```
收到 RESET → forwardReset → TaskConfigurator.handleReset()
→ selectiveClear() 清空 Redis（mapView/mapBlock/Car*/TaskConfig 全删）
→ 发送 TASK_READY 给 Controller
→ Controller 收到 TASK_READY → 启动 tick 循环
→ tick 循环发现没有 Car（已被删光）→ 空转
```

**根本原因**：RESET 只清空了数据，没有重新执行初始化。没有小车、没有障碍物、没有 TaskConfig，系统处于"激活但空"的状态。

### 解决方案

`TaskConfigurator.handleReset()` 需要做两件事：
1. 清空旧数据（当前已有）
2. **重新执行初始化**——用上一次的配置参数重新生成障碍物、放置小车、写入 TaskConfig

具体改动（`TaskConfiguratorMain.java`）：
- 增加一个 `Map<String, Object> lastConfig` 字段，在 `handleConfig` 时保存配置
- `handleReset` 改为：清除 → `initializer.initialize(bb, lastConfig)` → 发 TASK_READY

或者更简单的方案：RESET 后由前端自动发送一次 SET_CONFIG（带相同配置参数），让 TaskConfigurator 重新初始化。

## 问题 4：一键启动系统

### 现状

Launcher 模块（`launcher/.../LauncherMain.java`）就是设计来做这件事的，但当前实现中 Navigator/TargetPlanner/TaskConfigurator/Car 四个模块被标记为"跳过"。

### 解决方案

**前提**：第 2 轮审查中的 Fix 2 已经给四个模块添加了构造函数 + start() 方法，现在可以一键启动了。

**方式 A — 使用 Launcher（一键启动，同进程）**：

```powershell
.\mvnw.cmd exec:java -pl launcher "-Dexec.mainClass=com.substation.launcher.LauncherMain"
```

Launcher 在一个 JVM 内按顺序启动所有模块（每个在独立线程中）。各模块仍然是独立的逻辑单元，通过 MQ 和 Redis 通信，只是共享同一个 JVM 进程。

**方式 B — 写一个 .bat 脚本（一键启动，独立进程）**：

创建 `start_all.bat`：
```batch
@echo off
start "TaskConfigurator" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl task-configurator "-Dexec.mainClass=com.substation.taskconfigurator.TaskConfiguratorMain""
timeout /t 2 /nobreak >nul
start "Navigator" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl navigator "-Dexec.mainClass=com.substation.navigator.NavigatorMain""
start "TargetPlanner" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl target-planner "-Dexec.mainClass=com.substation.targetplanner.TargetPlannerMain""
timeout /t 1 /nobreak >nul
start "Car001" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl car "-Dexec.mainClass=com.substation.car.CarMain" "-Dexec.args=Car001""
start "Car002" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl car "-Dexec.mainClass=com.substation.car.CarMain" "-Dexec.args=Car002""
start "Car003" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl car "-Dexec.mainClass=com.substation.car.CarMain" "-Dexec.args=Car003""
timeout /t 1 /nobreak >nul
start "Display" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl display "-Dexec.mainClass=com.substation.display.DisplayMain""
timeout /t 1 /nobreak >nul
start "Controller" cmd /k "cd /d D:\car_homework && .\mvnw.cmd exec:java -pl controller "-Dexec.mainClass=com.substation.controller.ControllerMain""
echo 全部模块已启动，浏览器打开 http://localhost:8887
pause
```

**推荐先用方式 B**（bat 脚本），简单可靠，每个模块独立窗口，出问题容易排查。

## 问题 5：地图中红色块的含义

### 原因

是的，红色块（`#e94560`）代表**障碍物**。小车也是红色块是因为前端渲染逻辑中**小车和障碍物用了同一层绘制，颜色被覆盖了**。

当前 Canvas 五层渲染顺序：
```
L1 网格 → L2 探索区域 → L3 障碍物(红色) → L4 路径 → L5 小车(状态色圆点)
```

小车（L5）应该在障碍物（L3）之上绘制。如果小车所在格子恰好有障碍物标记，或者小车所在格子被 L3 的红色覆盖，就会出现"小车也是红色块"的视觉效果。

### 排查方向

1. 检查 `mapBlock` 中是否把小车位置也标记为障碍物（Car 模块移动时写 `setBlock(newPos, true)`）
2. 检查前端 `renderCars()` 是否在 `renderObstacles()` 之后调用
3. 检查车辆状态色是否被正确读取

**小车正常应该显示为**：实心圆 + 状态颜色外圈 + 白色编号文字。IDLE=灰圈、MOVING=蓝圈、READY=绿圈、BLOCKED=红圈（红圈 ≠ 红色方块）。

如果小车显示为实心红色方块而非带编号的圆圈，说明 L3 障碍物渲染覆盖了 L5 小车渲染，需要在前端 `app.js` 中调整渲染顺序或跳过有车格子的障碍物绘制。
