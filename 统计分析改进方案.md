# 统计分析模块 — 改进方案

> **用途**：指导 Person D / 前端优化 `analysis.html`、`analysis.js` 及仿真结束时的数据保存逻辑。  
> **现状文档**：`display/src/main/resources/web/统计分析模块设计文档.md`  
> **更新**：2026-06-22

---

## 1. 背景与目标

当前统计分析基于浏览器 `localStorage`（`sim_records`），在探索完成（100%）时由用户选择保存。列表页为卡片网格，详情页为 Canvas 柱状图 + 进度条。

**改进目标**：

1. 每张图表只表达**一个清晰问题**，避免「有图无故事」。
2. 突出本系统独有指标：**有效步数 / 探索效率 / 多车协作**。
3. 支撑答辩叙事：不仅「能跑完」，还要「跑得值、协作好、优化有效」。

---

## 2. 现有实现的问题

| 现有图表 | 问题 |
|----------|------|
| 各车总步数柱 + 各车有效步数柱 | 两根独立柱，**缺少对比关系**，信息重复 |
| 「关键指标对比」柱 | 将探索率%、有效率%、车辆数×10 画在同一纵轴 → **量纲混乱**，无法解读 |
| 探索进度条 | 保存时探索率已为 100% → **几乎永远满格**，信息量为零 |
| 数据维度 | 仅有**结束瞬间快照**，无过程曲线、无多局对比、无算法标签 |

**本系统真正的性能故事**（当前 UI 未充分表达）：

- 多车分布式协作探索
- **有效步数 / 总步数**（无效移动、绕路、重复巡逻）
- 加权路径 + 前沿目标分配（相对纯 BFS 的优化）
- 完成效率（tick、耗时、覆盖速度）

## 3. 设计原则

### 3.1 一张图一个问题

每张图配一句「读者问题」，例如：

| 图表 | 回答的问题 |
|------|------------|
| 探索效率环图 | 这次走得值不值？ |
| 堆叠柱（有效/无效） | 哪台车贡献大、谁在空转？ |
| 多局对比折线 | 优化后是否更少步数、更高效率？ |
| 探索率–tick 曲线 | 探索是前快后慢还是均匀推进？ |

### 3.2 指标定义（与代码一致）

| 指标 | 计算 | 数据来源 |
|------|------|----------|
| 总步数 | 各车 `steps` 之和 | `CarInfo.steps` / 保存时 `cars[]` |
| 有效步数 | 各车 `effectiveSteps` 之和 | `MoveExecutor` 写入 Redis |
| 探索效率 | `totalEffectiveSteps / totalSteps × 100%` | 保存记录时计算 |
| 无效步数 | `totalSteps - totalEffectiveSteps` | 派生 |
| 探索率 | 可探索格中已探索比例 | 任务完成时应为 100% |
| 耗时 | 仿真开始到结束秒数 | `app.js` 计时 |
| 节拍 | 结束时 `tick` | WebSocket `SimulationState` |

### 3.3 图表技术选型

- **短期**：继续纯 Canvas（与现网一致，无 CDN 依赖）。
- **中期**：若图表种类增多，可引入轻量库（如 Chart.js），非必须。

## 4. 推荐图表方案

### 4.1 ★★★ 探索效率仪表盘（详情页核心）

**表达**：本次仿真「有效移动占比」。

**形式**：

- 大号数字：**探索效率 XX%**（`有效/总`）
- 副指标：无效步数 = 总 − 有效
- **甜甜圈 / 环图**：绿色=有效步数，灰色=无效步数

**答辩话术**：「我们用有效步数衡量是否在外围重复巡逻；效率越高说明路径与目标分配越好。」

---

### 4.2 ★★★ 各车「有效 vs 无效」堆叠柱状图

**表达**：多车负载与无效巡逻分布。

**形式**（每台车一根柱，自下而上堆叠）：

```
Car001  [有效 ████████][无效 ██]
Car002  [有效 ██████  ][无效 ████]
```

**替换**：现有的「总步数柱 + 有效步数柱」两张独立图。

**答辩话术**：「可一眼看出分工是否均衡、是否存在单车空转。」

---

### 4.3 ★★★ 多局对比图（列表页 / 详情页）

**表达**：不同配置或优化前后的性能变化。

**形式**：

- 横轴：记录 #1、#2、#3（或日期）
- 纵轴系列（可选勾选）：
  - 总步数（越低越好）
  - 探索效率 %（越高越好）
  - 耗时 / tick（越低越好）

**数据前提**：保存记录时增加配置字段（见 §6）。

**答辩话术**：「加权路径 + 前沿目标后，同地图下总步数下降 X%，有效率提升 Y%。」

---

### 4.4 ★★☆ 负载均衡指数（卡片或小图）

**表达**：多车协作是否均衡。

**建议算法**（保存时写入记录）：

```
avg = 各车 effectiveSteps 平均值
std = 标准差
balanceScore = max(0, 1 - std/avg)   // 0~1，越高越均衡
```

或展示各车有效步数的 **迷你箱线图 / 极差**。

---

### 4.5 ★★☆ 完成效率指标

**表达**：整体探索「快不快」。

| 指标 | 公式 |
|------|------|
| 时间效率 | `totalEffectiveSteps / duration`（有效格/秒，近似） |
| 节拍效率 | `tick`（完成所用节拍数，可与车数、地图大小一起看） |

在详情顶栏与列表卡片展示，比单独罗列「356 步、122 秒」更直观。

---

### 4.6 ★★☆ 探索率–时间曲线（需扩展保存数据）

**表达**：探索过程节奏（前快后慢 / 均匀）。

**形式**：折线图，横轴 `tick`，纵轴 `explorationRate %`。

**实现思路**：

- 任务进行中每 N 拍采样一次（如每 10 tick），或结束时从 `explorationEvents` / 回放数据重采样 15~20 点。
- 写入 `sim_records[].rateTimeline: [{tick, rate}, ...]`。

**价值**：过程曲线比终点四个数字更有说服力，答辩演示效果好。

---

### 4.7 ★☆☆ 热力图缩略图（可选）

**表达**：哪些区域被反复访问（无效探索高发区）。

**数据**：保存时附带 `mapHeat` 摘要（Base64 位图或 Top-N 热点坐标）。

**与系统故事**：闭环「加权路径减少绕路」；实现量大于前几项。

---

### 4.8 ★☆☆ 算法 / 策略 A–B 对比（专项视图）

**表达**：BFS vs 加权路径、不同车数/障碍率对比。

**形式**：同一页面并排两张记录卡片，或表格：

| 指标 | 纯 BFS | 加权+前沿 |
|------|--------|-----------|
| 总步数 | 848 | 612 |
| 有效率 | 31% | 44% |
| 耗时 | 180s | 132s |

**数据**：依赖保存 `algorithm`、`obstacleRatio`；对比测试逻辑见 `ExplorationEfficiencyComparisonTest`。

## 5. 建议的详情页布局

```
┌──────────────────────────────────────────────────────────────┐
│  探索效率 72%  │  总步 356  │  有效 264  │  耗时 122s  │ tick 245 │
├────────────────────────────┬─────────────────────────────────┤
│  环图：有效 / 无效           │  堆叠柱：各车 有效 + 无效         │
├────────────────────────────┴─────────────────────────────────┤
│  探索率–tick 曲线（有数据时）  或  与历史记录对比折线（≥2 条时）   │
└──────────────────────────────────────────────────────────────┘
```

**建议删除**：

- 「关键指标对比」混轴柱状图
- 任务完成后的重复「探索进度条」（100% 无信息量）

## 6. 列表页改进

### 6.1 卡片信息层级

当前：探索率大号 + 步数/耗时/车数一行小字。

**建议**：

```
仿真记录 #3
2026-6-22 15:30
┌─────────────────┐
│  有效率 72%      │  ← 主视觉（颜色：>60% 绿，40–60% 黄，<40% 红）
└─────────────────┘
5 车 · 122s · BFS · 障碍 15%
```

### 6.2 列表顶栏汇总（可选）

- 最近 N 局平均有效率
- 最佳 / 最差记录高亮

## 7. 保存数据结构扩展（`sim_records`）

在现有 JSON 上**增量字段**（旧记录兼容：缺字段时用 `--` 或隐藏对应图）：

```json
{
  "explorationRate": 100,
  "tick": 245,
  "duration": 122,
  "totalSteps": 356,
  "totalEffectiveSteps": 264,
  "efficiencyPercent": 74,
  "wastedSteps": 92,
  "carCount": 5,
  "algorithm": "BFS",
  "obstacleRatio": 0.15,
  "mapWidth": 30,
  "mapHeight": 30,
  "balanceScore": 0.82,
  "cars": [
    { "carId": "Car001", "steps": 80, "effectiveSteps": 62, "status": "IDLE" }
  ],
  "rateTimeline": [
    { "tick": 0, "rate": 12 },
    { "tick": 50, "rate": 45 },
    { "tick": 245, "rate": 100 }
  ],
  "timestamp": 1718000000000,
  "date": "2026-6-22 15:30:22"
}
```

**写入位置**：`display/.../js/app.js` 的 `showSavePopup` 保存逻辑；配置从 `liveData.taskConfig` 与当前表单读取。

## 8. 实施分期

| 阶段 | 内容 | 工作量 | 答辩价值 |
|------|------|--------|----------|
| **P0** | 环图（有效/无效）+ 堆叠柱；删混轴图与重复进度条 | 小（1 人天） | 高 |
| **P0** | 保存时写入 `algorithm`、`obstacleRatio`、`efficiencyPercent` | 很小 | 高 |
| **P1** | 多局对比折线/柱（≥2 条 `sim_records`） | 中 | 很高 |
| **P1** | `rateTimeline` 采样 + 探索率曲线 | 中 | 很高 |
| **P2** | `balanceScore`、列表效率徽章与配色 | 中 | 中高 |
| **P2** | `mapHeat` 缩略图 | 中偏大 | 中 |
| **P3** | 对接 `AnalysisEngine` + SQL 持久化、导出 CSV | 大 | 长期 |

## 9. 答辩叙事模板

> 本系统是多进程黑板架构下的**多车协作探索**。除覆盖率和总步数外，我们引入**有效步数**衡量是否重复踩已探索区域。统计分析模块展示：
>
> 1. **探索效率**（有效/总）— 路径与目标质量；  
> 2. **各车有效/无效堆叠** — 协作负载；  
> 3. **多局 / 多算法对比** — 加权路径与前沿分配带来的步数与效率提升。
>
> 数据在任务完成时由用户保存，支持多次实验对比，验证优化策略有效。
