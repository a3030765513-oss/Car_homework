## 三、Person A：common + controller

### 为什么这样分配

common 是所有人依赖的公共库，controller 是唯一调度器，这两个模块天然需要一个人同时掌控，否则接口定义来回沟通成本太高。

### 3.1 common 模块交付清单

| 文件 | 职责 |
|------|------|
| `common/Point.java` | 坐标类（x, y）+ 曼哈顿距离 + JSON 序列化 |
| `common/CarStatus.java` | 5 状态枚举（IDLE/WAITING_ROUTE/READY/MOVING/BLOCKED）+ 颜色 + 中文名 |
| `common/AlgorithmType.java` | BFS/ASTAR 枚举 |
| `common/RouteStep.java` | 路径步骤（position + stepIndex） |
| `common/SimulationState.java` | 全局状态快照（含内部类 CarInfo），WebSocketBridge 每节拍构建前端 JSON 的数据载体 |
| `common/BlackboardClient.java` | Redis 封装，9 个 key 的全部读写方法（见开发计划 §2.2.2） |
| `common/DistributedLock.java` | Redis 分布式锁（SETNX + Lua 释放，超时 5s） |
| `common/MessageBus.java` | RabbitMQ 封装（声明队列、发布、订阅、连接管理） |
| `common/MessageTypes.java` | 所有消息类型常量 |
| `common/MessageBuilder.java` | 统一消息 JSON 构建工具（fastjson2） |

**对外接口（供 B/C/D 使用）**：

```java
// BlackboardClient — 所有知识源通过它读写 Redis
BlackboardClient bb = new BlackboardClient("localhost", 6379);
bb.setCarStatus("Car001", CarStatus.IDLE);
Point pos = bb.getCarPosition("Car001");

// DistributedLock — Car 移动时使用
DistributedLock lock = new DistributedLock(jedis, "lock:Car001");
if (lock.tryLock(5000)) { ... lock.unlock(); }

// MessageBus — 所有模块通过它收发 MQ 消息
MessageBus mb = new MessageBus("localhost", 5672, "guest", "guest");
mb.declareCarQueue("Car001");
mb.subscribe("Car_Car001", msg -> handleMessage(msg));
mb.publish("ControllerCmd", messageJson);
```

### 3.2 controller 模块交付清单

> ⚠️ **任务书约束**：控制器必须独立运行，且在操作系统中只能运行一个实例。ControllerMain 启动时需通过 Redis SETNX `controller:instance` 互斥锁确保单实例。

| 文件 | 职责 |
|------|------|
| `controller/ControllerMain.java` | 入口：连接 Redis + RabbitMQ，检测单实例（SETNX `controller:instance`），启动 TickScheduler |
| `controller/TickScheduler.java` | 定时器驱动 tick() 循环，默认 500ms 间隔 |
| `controller/StatusDispatcher.java` | 遍历所有已注册车辆（KEYS Car*:Status），按状态分发命令（核心调度逻辑） |
| `controller/CommandHandler.java` | 订阅 ControllerCmd，处理知识源回复（TARGET_ASSIGNED / ROUTE_PLANNED / MOVED / BLOCKED / ROUTE_DONE / TASK_READY）+ Web 命令转发 |

**Controller 对外的接口（MQ 消息）**：

```
发送队列：
  → TargetPlannerCmd: ASSIGN_TARGET {carId, tick}
  → NavigatorCmd:     PLAN_ROUTE {carId, start{x,y}, target{x,y}, algorithm, tick}
  → Car_CarID:        TICK_MOVE {tick} | BLOCKED_TIMEOUT {carId}
  → TaskConfigCmd:    FORWARD_CONFIG {config...} | FORWARD_RESET {}
  → UpdateView(fanout): REFRESH_ALL {tick, explorationRate}

接收队列：
  ← ControllerCmd: TARGET_ASSIGNED | ROUTE_PLANNED | MOVED | BLOCKED | ROUTE_DONE | TASK_READY | SET_CONFIG | RESET
```

2. ControllerMain + CommandHandler（先能启动 + 接收消息）
3. TickScheduler + StatusDispatcher（核心调度逻辑）
4. 与 B/C/D 联调

## 四、Person B：car + target-planner

### 为什么这样分配

car 和 target-planner 都围绕"小车"展开：一个是小车的执行端（移动/状态机），一个是为小车分配目标的策略端（贪心算法）。同一个人负责可以在目标分配策略和移动执行之间快速迭代。

### 4.1 car 模块交付清单

| 文件 | 职责 |
|------|------|
| `car/CarMain.java` | 入口：接受 carId 命令行参数；连接 Redis + RabbitMQ，订阅 Car_CarID，创建 CarAgent |
| `car/CarAgent.java` | 消息分发：收到 TICK_MOVE → 调用 MoveExecutor |
| `car/MoveExecutor.java` | 8 步原子移动操作（加锁→读状态→peek路径→检查障碍→pop路径→更新位置→点亮3×3→写状态→解锁） |

**Car 对外的接口（MQ 消息）**：

```
接收队列：
  ← Car_CarID: TICK_MOVE {tick} | BLOCKED_TIMEOUT {carId}

发送队列：
  → ControllerCmd: MOVED {carId, newPosition{x,y}, routeRemaining} 
  → ControllerCmd: ROUTE_DONE {carId, finalPosition{x,y}}
  → ControllerCmd: BLOCKED {carId, blockedPosition{x,y}, blockedTick}
```

**Car 内部状态转换规则**（只写自己管辖的变迁）：

```
READY → MOVING   收到 TICK_MOVE，下一步无障碍
MOVING → READY   移动成功，路径仍有剩余
MOVING → IDLE    路径走完，发 ROUTE_DONE
MOVING → BLOCKED 下一步有障碍，发 BLOCKED
(初始化时) → IDLE  注册到黑板
```

> ⚠️ Car 永不写 WAITING_ROUTE（那是 Controller 写的）

> 📌 **动态添加小车**：CarMain 接受 carId 参数（如 `java -jar car.jar Car006`），启动即自动注册到黑板。Controller 通过 Redis `KEYS Car*:Status` 动态发现已注册车辆，新进程启动即加入，无需改配置。

### 4.2 target-planner 模块交付清单

| 文件 | 职责 |
|------|------|
| `target-planner/TargetPlannerMain.java` | 入口：订阅 TargetPlannerCmd |
| `target-planner/GreedyTargetAllocator.java` | 贪心分配算法（读未探索区域 → 排序 → 按距离规则分配 → 写 CarID:Target） |

**TargetPlanner 对外的接口（MQ 消息）**：

```
接收队列：
  ← TargetPlannerCmd: ASSIGN_TARGET {carId, tick}

发送队列：
  → ControllerCmd: TARGET_ASSIGNED {carId, target{x,y}, success}
```

> ⚠️ TargetPlanner 只写 CarID:Target（领域数据），不写 CarID:Status

### 4.3 开发顺序

1. CarMain + 基本消息收发
2. MoveExecutor（核心：8 步原子操作 + 分布式锁）
3. CarAgent（状态机逻辑）
4. TargetPlannerMain + GreedyTargetAllocator
5. 与 Person A 的 Controller 联调

## 五、Person C：navigator + task-configurator

### 为什么这样分配

navigator 是纯算法模块，task-configurator 是纯初始化模块，两者都偏"离线处理"而非实时调度。同一个人可以先把简单的 task-configurator 做完跑通，再集中精力写算法。

### 5.1 navigator 模块交付清单

| 文件 | 职责 |
|------|------|
| `navigator/NavigatorMain.java` | 入口：订阅 NavigatorCmd |
| `navigator/PathPlanner.java` | 接口：`List<Point> plan(Point start, Point target, BlackboardClient bb)` |
| `navigator/BfsPathFinder.java` | BFS 实现（4 方向，避开障碍+车辆，边界检查） |
| `navigator/AStarPathFinder.java` | A* 实现（曼哈顿距离启发，PriorityQueue 开放列表） |
| `navigator/PathPlannerFactory.java` | 工厂：根据 algorithm 参数返回 BFS 或 A* 实例 |

**Navigator 对外的接口（MQ 消息）**：

```
接收队列：
  ← NavigatorCmd: PLAN_ROUTE {carId, start{x,y}, target{x,y}, algorithm, tick}

发送队列：
  → ControllerCmd: ROUTE_PLANNED {carId, routeFound, routeLength}

黑板写入：
  → CarID:RouteList (LPUSH 批量写入路径坐标 JSON)
```

> ⚠️ Navigator 只写 CarID:RouteList（领域数据），不写 CarID:Status

> 📌 **Navigator 多实例支持**（任务书要求）：Navigator 共享 `NavigatorCmd` Classic Queue（非独占），可同时启动多个 Navigator 进程竞争消费。RabbitMQ 默认轮询分发，天然实现"分解导航压力"。验收方式：`NavigatorMain` 多次启动，RabbitMQ 管理界面可看到多个消费者共享同一队列。

### 5.2 task-configurator 模块交付清单

| 文件 | 职责 |
|------|------|
| `task-configurator/TaskConfiguratorMain.java` | 入口：订阅 TaskConfigCmd |
| `task-configurator/TaskInitializer.java` | 初始化流程：清空 Redis → 随机障碍物 → 分配小车初始位置 → 点亮初始区域 → 写 TaskConfig |

**TaskConfigurator 对外的接口（MQ 消息）**：

```
接收队列：
  ← TaskConfigCmd: FORWARD_CONFIG {mapWidth, mapHeight, carCount, obstacleRatio, algorithm, tickInterval}
                   FORWARD_RESET {}

发送队列：
  → ControllerCmd: TASK_READY {}

黑板写入：
  → mapView, mapBlock, CarID:Position, CarID:Status, CarID:Steps, TaskConfig (全部 9 个 key 的初始化)
```

### 5.3 开发顺序

1. TaskConfiguratorMain + TaskInitializer（先跑通初始化流程）
2. NavigatorMain + 消息收发
3. BfsPathFinder（先实现，可联调）
4. AStarPathFinder + PathPlannerFactory
5. 与 Person A 的 Controller 联调

## 八、各模块间的接口契约

以下是每个人都必须遵守的、跨模块的接口约定。写完 common 后，这些由 common 模块的类型和常量保证。

### 8.1 Redis Key 命名规范

| Key 模板 | 类型 | 写入者 | 读取者 |
|----------|------|--------|--------|
| `mapView` | bitmap | Car, TaskConfigurator | 所有人 |
| `mapBlock` | bitmap | Car, TaskConfigurator | 所有人 |
| `{carId}:Position` | hash `{x, y}` | Car, TaskConfigurator | 所有人 |
| `{carId}:Target` | hash `{x, y}` | TargetPlanner (写), Controller (删) | 所有人 |
| `{carId}:RouteList` | list (JSON string) | Navigator (LPUSH), Car (RPOP) | 所有人 |
| `{carId}:Status` | string | Controller, Car (各自管辖的变迁) | 所有人 |
| `{carId}:Steps` | string (int) | Car | 所有人 |
| `{carId}:BlockedTick` | string (int) | Car (写), Controller (读/删) | Controller, Display |
| `TaskConfig` | hash | TaskConfigurator | 所有人 |
| `lock:{carId}` | string (SETNX) | Car (加锁), — (自动过期) | — |

### 8.2 Car 编号约定

车辆 ID 格式为 `Car` + 3 位数字（如 `Car001`）。Controller 和 TargetPlanner 通过 Redis `KEYS Car*:Status` 动态获取当前已注册车辆列表，而非从 `carCount` 硬推导。这确保了新 CarMain 进程启动后自动加入，无需修改配置。

```java
// Controller 动态发现已注册车辆
Set<String> carIds = jedis.keys("Car*:Status").stream()
    .map(key -> key.replace(":Status", ""))
    .collect(Collectors.toSet());
```

前端显示编号从 carId 提取数字部分（`Car001` → `001`），不预设数量。

### 8.3 消息格式统一规范

所有 MQ 消息 body 为 fastjson2 序列化的 JSON，结构：

```json
{
  "type": "消息类型",
  "tick": 0,
  "carId": "Car001",
  "timestamp": 1717400000000,
  "data": { }
}
```

### 8.4 状态写入权限矩阵

| 状态变迁 | 写入者 | 触发条件 |
|----------|--------|----------|
| → IDLE | Car | 初始化注册 / 路径走完 / 受阻超时后 Controller 写入 |
| → IDLE | Controller | BLOCKED 超时 ≥ 2 节拍 |
| → WAITING_ROUTE | Controller | 收到 TARGET_ASSIGNED，carId 在 pendingTargetRequests 中且 target 非空 |
| → READY | Controller | 收到 ROUTE_PLANNED 且 routeFound=true |
| → READY | Car | 移动一步完成，路径非空 |
| → MOVING | Car | 收到 TICK_MOVE，下一步无障碍 |
| → BLOCKED | Car | 下一步有障碍 |

### 8.5 队列名称常量

```java
public class QueueNames {
    public static final String CAR_PREFIX = "Car_";
    public static final String NAVIGATOR_CMD = "NavigatorCmd";
    public static final String TARGET_PLANNER_CMD = "TargetPlannerCmd";
    public static final String TASK_CONFIG_CMD = "TaskConfigCmd";
    public static final String CONTROLLER_CMD = "ControllerCmd";
    public static final String UPDATE_VIEW_EXCHANGE = "UpdateView";
    
    public static String carQueue(String carId) { return CAR_PREFIX + carId; }
}
```

### 8.6 分布式锁约定

```java
// 锁 key 格式
String lockKey = "lock:" + carId;

// 加锁：SETNX + 超时 5 秒
boolean acquired = lock.tryLock(5000);

// 释放：Lua 脚本验 ownership
lock.unlock();

// Car 移动期间的复合操作必须加锁
// Controller 写 Status 变迁前应尝试加锁（乐观锁 WATCH 或直接 tryLock 短超时）
```

## 十、沟通与协作规则

| 规则 | 说明 |
|------|------|
| common 接口定稿 | **common 接口设计（BlackboardClient 方法签名、MessageBus 方法签名、消息格式）在第 1 天完成初版后，全员 review 确认方可定稿** |
| common 接口变更 | 变更必须先群内说明动机和影响范围，获全员确认后再改 |
| Redis key 增删 | 必须全员同意，更新本文件 §8.1 |
| MQ 消息格式变更 | 必须全员同意，更新本文件 §8.3 |
| 每个人的模块 | 独立进程，自己写 Main 入口，自己加单元测试 |
| 代码仓库 | 分支命名 `feature/common`、`feature/controller`、`feature/car`、`feature/navigator`、`feature/display` 等；common 先合入 main → 其他人 rebase main 后再开发；提 PR 前必须本地 `mvn test` 通过 |
| 每日对齐 | 收工前 15 分钟，各自说进度 + 阻塞项 |
| 互为 backup | 同组另外三人中至少一人能协助排查对方模块的问题 |
| 仓库地址 | **【待填写】** Git 仓库 URL |

## 十一、风险预案

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| common 延期 | 中 | 阻塞所有人 | B/C/D 第 1 天不等待，直接开工：搭建 Maven 模块骨架、定义 Main 入口空壳、学习 Jedis/RabbitMQ API、Person D 写前端静态页面（不依赖 Java 后端）；A 优先交付 BlackboardClient + MessageBus 核心 2 类，其余辅助类可延后 |
| 某人模块卡住 | 中 | 该模块进度滞后 | 提前约定"互为 backup"（见 §十）；先推进模块的降级版本（见开发计划 §七 降级路径），先跑通再完善 |
| Redis/RabbitMQ 环境问题 | 低 | 无法联调 | 统一使用 docker-compose.yml 一键启动（见开发计划 §0.2），禁止各人自行安装不同版本 |
| Controller 异步调度 bug | 中 | 车状态卡死 | 优先用状态日志验证（每 tick 打印所有车状态）；加单元测试覆盖 5 种状态分支 |
| 联调时接口不匹配 | 中 | 联调阻塞 | common 消息格式初版定稿后不再改；联调前各自用单元测试 mock MQ 消息验证 |
